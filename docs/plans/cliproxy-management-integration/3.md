# Milestone 3 — Dynamic Model Discovery

## Scope

Wire the `discoverModels()` function from the API client into the application startup and modal open flows so that the local provider's model list is dynamically populated from the CLIProxy config endpoint. Models are discovered:

1. At app startup (in `RoutingManager.initialize()`) — fire-and-forget async
2. On every `renderProviderCards()` call (in `loadCliProxySettings()`) — extends the method added in Milestone 2. This fires on modal open AND on every internal re-render (add model, remove model, etc.)

The `VITE_CLIPROXY_MODELS` env var remains as the fallback when the proxy is unreachable or returns no models. Discovered models are transient (in-memory only, never persisted to localStorage).

## Implementation Notes

- `RoutingManager.providerManager` is a private field (line 24), so `discoverCliProxyModels()` can access it directly within the class — no getter needed
- `RoutingManager.updateAvailableModels()` is also private (line 72) — called directly within the class
- In `ProviderManagementUI.loadCliProxySettings()`, after calling `this.providerManager.updateLocalModels(models)`, also call `this.notifyModelsChanged()` — this invokes the `onModelsChangedCallback` (line 543) which is wired to `RoutingManager.refreshProviders()` (line 47-49)
- The `#cliproxy-models-status` text should show "Using models from environment" when discovery returns empty (not "0 models" which would be confusing alongside env-var models that ARE present). Don't call `updateLocalModels` in that case to preserve env-var fallback.
- When proxy is unreachable (both `fetchConfig` returns empty config AND `fetchQuotaExceededSettings` returns `null`), do NOT update `#cliproxy-models-status` — the M2 unreachable notice already communicates the problem.
- **No infinite loop concern:** `loadCliProxySettings()` calls `notifyModelsChanged()` → `refreshProviders()` (RoutingManager line 89) → `updateAvailableModels()` + `updateTriggerState()`. Critically, `refreshProviders()` does NOT call `renderProviderCards()`, so model discovery within `loadCliProxySettings()` will never trigger another `renderProviderCards()` → `loadCliProxySettings()` cycle.
- Since `loadCliProxySettings()` now runs on every `renderProviderCards()` call (not just modal open), model discovery runs more often. This is acceptable: the fetches are small GET requests to localhost with 5s timeout, and the fire-and-forget pattern means they don't block the UI.
