# Milestone 1 — CLIProxy Management API Client

## Scope

Create a standalone, stateless API client module (`Routing/CLIProxyManagementClient.ts`) that wraps all CLIProxy Management API interactions used by this integration. This module has zero dependencies on other Routing modules — it only uses browser `fetch` and `import.meta.env`.

Includes:
- TypeScript interfaces for all request/response shapes (quota settings, config, key entries, openai-compatibility entries, claude-api-key entries with their model arrays)
- Helper functions: `getManagementBaseURL()`, `getAuthHeaders()`, `isCliProxyConfigured()`
- Individual fetch wrappers for quota-exceeded GET/PUT endpoints: `getQuotaSwitchProject()`, `setQuotaSwitchProject()`, `getQuotaSwitchPreviewModel()`, `setQuotaSwitchPreviewModel()`
- Combined `fetchQuotaExceededSettings()` that fetches both toggles in parallel
- `fetchConfig()` for `GET /config`
- Pure `discoverModels(config: CLIProxyConfig)` that extracts model IDs from config (no I/O, unit-testable)
- All fetch calls use `AbortController` with a 5-second timeout
- Auth via `VITE_CLIPROXY_API_KEY` env var as `Authorization: Bearer` header (accessed via `import.meta.env.VITE_CLIPROXY_API_KEY`)
- In dev mode (`import.meta.env.DEV`): requests route through `/cliproxy/v0/management` (Vite proxy)
- In production: requests go to `${import.meta.env.VITE_CLIPROXY_BASE_URL}/v0/management` directly

**Important implementation notes:**
- Follow the dev/prod URL pattern from `AIProvider.ts:563-566` — check `import.meta.env.DEV` to decide the base URL
- The Vite proxy at `/cliproxy` strips the prefix (`vite.config.ts:23`), so `/cliproxy/v0/management/config` → `/v0/management/config` at the target
- `codex-api-key` entries in the config do NOT have `models` arrays (only `excluded-models`) — do not attempt to extract models from them
- `claude-api-key` entries have `models[]` with `{name, alias}` objects (same structure as openai-compatibility models)
- The GET response for quota endpoints uses the setting name as key (e.g., `{"switch-project": true}`), not `{"value": true}` — parse accordingly
- `fetchQuotaExceededSettings()` must NOT delegate to the individual `getQuotaSwitchProject()`/`getQuotaSwitchPreviewModel()` functions (those swallow errors and return `false`). Instead, it should make its own fetch calls so it can detect network errors and return `null` to distinguish "proxy unreachable" from "both toggles are false"
- All `AbortController` timeouts must be cleared with `clearTimeout(timeoutId)` after a successful fetch response to prevent calling `abort()` on already-completed requests
- A private helper like `managementFetch(path: string, options?: RequestInit): Promise<Response>` is encouraged to reduce duplication of AbortController/timeout/auth logic across the 7 fetch call sites. If used, ensure the helper still propagates errors (does not catch) so each caller controls its own error handling.
- `discoverModels()` must guard against entries where `models` is `undefined` or missing — not all `openai-compatibility` or `claude-api-key` entries have a `models` array (e.g., an entry might only define `excluded-models`). Use optional chaining or nullish checks: `entry.models?.forEach(...)` or `(entry.models || [])`
